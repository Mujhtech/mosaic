# Mosaic Protocol 0.2 RC4

## Status and authority

Protocol `0.2` RC4 is a release candidate, not an approved contract. RC4
supersedes the unapproved RC3 candidate. Flutter, SwiftUI, Jetpack Compose,
Studio, Local Preview, accessibility, fallback, and product review gates must
all pass before approval. Pre-release Protocol and Local Preview `0.1` have been
retired; `0.2` is the only active contract.

Canonical artifacts:

- `protocol/schema/v0.2/paywall.schema.json`
- `protocol/compatibility/v0.2.json`
- `protocol/fixtures/v0.2/complete-paywall.json`
- `protocol/fixtures/v0.2/edge-cases.json`
- `protocol/fixtures/v0.2/invalid/`

The document is closed and declarative. Unknown properties, missing or cyclic
references, unsafe URLs, executable content, and arbitrary expressions reject
the complete document atomically.

## Document design system

Every document contains three bounded catalogs. Empty catalogs are valid; each
catalog accepts at most 256 entries.

```json
{
  "designSystem": {
    "colors": [
      { "id": "brand-primary", "name": "Brand primary", "value": "#007F73FF" },
      {
        "id": "brand-accent",
        "name": "Brand accent",
        "value": { "type": "colorToken", "id": "brand-primary" }
      }
    ],
    "backgrounds": [],
    "shadows": []
  }
}
```

Token IDs and display names are unique inside their category. Colour,
background, and shadow references resolve only inside their own category.
Token-to-token references are allowed, but every reference must resolve and
the graph must be acyclic. Renderers resolve tokens from the accepted document;
there is no server or project design-library lookup at render time. Editing an
inline value after choosing a token is an authoring concern: Studio detaches
the field from the token instead of mutating the token silently.

## Colours, backgrounds, and shadows

A `Color` is one of the existing semantic names, canonical uppercase
`#RRGGBBAA`, or a strict token reference:

```json
{ "type": "colorToken", "id": "brand-primary" }
```

A `Background` is exactly one of:

```json
{ "type": "color", "value": "surface.default" }
```

```json
{
  "type": "linearGradient",
  "angle": 135,
  "stops": [
    { "position": 0, "color": "surface.default" },
    { "position": 1, "color": { "type": "colorToken", "id": "brand-primary" } }
  ]
}
```

```json
{
  "type": "radialGradient",
  "center": { "x": 0.5, "y": 0.25 },
  "radius": 1,
  "stops": [
    { "position": 0, "color": "surface.elevated" },
    { "position": 1, "color": "transparent" }
  ]
}
```

```json
{
  "type": "image",
  "assetId": "offer-texture",
  "contentMode": "fill",
  "fallbackColor": "surface.default"
}
```

```json
{
  "type": "video",
  "assetId": "offer-video",
  "posterAssetId": "offer-poster",
  "contentMode": "fill",
  "fallbackColor": "surface.default"
}
```

```json
{ "type": "backgroundToken", "id": "offer-gradient" }
```

Gradients contain two through eight stops. Positions are in `0...1`, strictly
increasing, and unique. Linear angle is a physical canvas direction from the
first stop to the last: `0°` is left-to-right, `90°` is top-to-bottom, angles
increase clockwise, and RTL never mirrors the angle. The accepted range is
`0...360`, where `360°` is equivalent to `0°`. Radial center coordinates
are normalized `0...1`; radius is positive and at most `2`.

Media backgrounds are decorative: they never expose accessibility content or
controls. Video is always muted, autoplaying, looping, and control-free. When
video cannot load, the renderer uses its declared poster if that image is
available, then the declared fallback colour. An unavailable image uses its
fallback colour. Every failure emits a safe diagnostic and never crashes the
host.

Eligible authored boxes accept one shadow:

```json
{
  "type": "shadow",
  "color": "#00000033",
  "offsetX": 0,
  "offsetY": 8,
  "blurRadius": 24
}
```

or `{ "type": "shadowToken", "id": "floating" }`. RC4 has no spread,
blend mode, backdrop blur, multiple shadows, or ordered effect stack. Product
Card and Product Badge Default/Selected box styles use this same shadow model.

## Assets

Image and video assets use a bundled logical key or a safe absolute HTTPS URL:

```json
{
  "type": "image",
  "id": "offer-poster",
  "source": { "type": "remote", "url": "https://assets.example.com/poster.webp" },
  "fallback": {
    "type": "placeholder",
    "value": {
      "default": "Image unavailable",
      "localizationKey": "paywall.assets.poster_fallback"
    }
  }
}
```

```json
{
  "type": "video",
  "id": "offer-video",
  "source": { "type": "bundled", "key": "mosaic.offer.video" }
}
```

Remote URLs must be absolute HTTPS, at most 2,048 characters, and contain no
credentials, control characters, malformed hosts, or invalid ports. An Image
component references only an image asset. Image/video backgrounds and poster
references must match their declared asset type.

## Uniform sizing

Every eligible authored box uses one optional uniform sizing object:

```json
{
  "sizing": {
    "width": "fill",
    "height": { "mode": "fixed", "value": 180 }
  }
}
```

Both axes are required when `sizing` is present. Each axis is `"fit"`,
`"fill"`, or `{ "mode": "fixed", "value": positiveLogicalSize }`.
Candidate value `"content"` is replaced by `"fit"`.

Stack, Text, Image, Icon, Feature List, Product Selector, Product Card,
Product Badge, Button, Carousel, Switch, and Countdown are eligible. Screen
Scroll Container is viewport-owned and excluded. Image may retain an optional
positive `aspectRatio` as intrinsic-ratio guidance; it has no separate width or
height field.

Fixed boxes clip visual overflow while the full semantic value remains
available to assistive technology. Fill requires a bounded parent axis. If the
axis is unbounded, the renderer uses Fit and emits `layout.unboundedFill`
instead of failing or producing infinite layout.

## Screen and Sheet presentation

Every screen declares presentation explicitly:

```json
{ "presentation": { "type": "screen" } }
```

or:

```json
{ "presentation": { "type": "sheet" } }
```

`initialScreenId` must reference a Screen. A Sheet presents on the native modal
surface over the most recent Screen in navigation history, remains within the
safe viewport, and scrolls internally through its existing Scroll Container.
`navigateBack` dismisses/pops it through the same history model. Converting a
Screen to Sheet or Sheet to Screen changes only `presentation`; ID, content,
actions, and references remain unchanged.

RC4 has no detents, draggable sheet handle, nested presentation options,
custom transition, or arbitrary screen position. The existing rules remain:
one through ten screens, globally unique component IDs, complete reachability,
screen-local purchase/Switch references, and an acyclic `navigateTo` graph.

## Components, actions, and product structure

RC4 retains RC3 component and action semantics:

- generalized vertical/horizontal Stack;
- Text, Image, Icon, Feature List, Carousel, Switch, and Countdown;
- one composable Button with Purchase, Restore, Close, Navigate To, Navigate
  Back, or Open External URL;
- Product Selector with ordered authored Product Cards and optional Product
  Badge content;
- complete Default plus recursively partial Selected box styles; and
- closed `{{ product.name }}` / `{{ product.price }}` substitution inside
  Product Card content and accessibility copy only.

No remote executable behavior, arbitrary interpolation, or platform resource
identifier is part of the protocol.

## Capabilities and fallback

RC4 adds exact capabilities derived only when the corresponding feature occurs:

```text
layout.heightSizing       navigation.sheets
style.designTokens        style.gradientBackground
style.mediaBackground     style.shadow
asset.remoteImage         asset.bundledVideo
asset.remoteVideo
```

Existing `asset.bundledImage`, styling, component, action, localization,
accessibility, and runtime capabilities remain exact. A client missing any
required capability rejects the whole candidate before replacing its last
accepted document. There is no partial style or component rendering.

Runtime fallback is separate from compatibility rejection: unavailable media
uses poster/colour fallback and diagnoses; Fill on an unbounded axis uses Fit
and diagnoses; root Navigate Back remains a no-op diagnostic. Failure still
resolves through last accepted, bundled fallback, then configuration
unavailable.

## Explicit deferrals

RC4 intentionally defers multiple fills/effects/shadows, video audio or
controls, arbitrary animation, freeform canvas coordinates in the document,
custom sheet transitions/detents, and project/server design libraries. Those
features require a later reviewed protocol change.

Migration and Local Preview behavior are documented in
`local-preview-v0.2.md` and `versioning.md`.
