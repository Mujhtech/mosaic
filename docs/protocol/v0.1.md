# Mosaic Protocol 0.1

## Status and scope

Protocol `0.1` is a Phase 0 draft. It proves that every first-party protocol
reader can model the same minimal paywall document. It does not define a full
renderer or the broader component catalog planned for later phases.

The canonical artifacts are:

- schema: `protocol/schema/v0.1/paywall.schema.json`
- compatibility manifest: `protocol/compatibility/v0.1.json`
- fixture: `protocol/fixtures/v0.1/minimal-paywall.json`

The fixture is the single source of truth. Tests and examples may package it as
a build resource, but they must source it from that path and must not maintain a
divergent fixture.

## Document contract

A `0.1` document is a closed JSON object with these required fields:

| Field           | Meaning                                                               |
| --------------- | --------------------------------------------------------------------- |
| `schemaVersion` | Exact protocol identifier. It must be `0.1`.                          |
| `id`            | Stable document identifier.                                           |
| `revision`      | Draft revision integer from 1 through 2,147,483,647.                  |
| `compatibility` | Exact capabilities required to understand the document.               |
| `layout`        | One root vertical layout.                                             |

Every object rejects unknown properties. The document contains declarative data
only and has no field for executable code or unrestricted expressions.

`revision` follows JSON Schema's mathematical integer semantics. A JSON number
with an integral value, including a spelling such as `1.0`, is valid within the
documented range. Non-finite runtime values are not JSON numbers and must be
rejected by protocol readers. Publication immutability is enforced outside this
schema.

## Shared text value

Every visible string in `0.1` uses this shape:

```json
{
  "default": "Continue",
  "localizationKey": "paywall.purchase"
}
```

A reader first attempts to resolve `localizationKey` from its supplied
localization source. If the key is unavailable, it displays `default`. Protocol
`0.1` does not define localization catalogs, interpolation, plural rules, or
directional layout.

## Layout semantics

The root layout has `type: "vertical"` and renders children in document order
along its primary vertical axis.

- `padding` applies the same nonnegative logical spacing to all four content
  edges.
- `gap` applies nonnegative logical spacing between adjacent children, never
  before the first child or after the last child.
- Logical spacing is interpreted in the reader's native density-independent
  layout units; it is not a physical-pixel contract.

Nested layouts, scrolling, alignment, safe areas, fixed sizing, and styling are
outside `0.1`.

## Component semantics

### `text`

Displays its localized `value` as ordinary content. Typography and alignment
are reader defaults in `0.1`.

### `featureList`

Presents one or more localized feature items in source order. Visual markers
are a renderer concern; item order and text are the shared semantic contract.

### `productSelector`

Declares one or more provider-independent `productId` references and one
`initiallySelectedProductId`. Product IDs must be unique, and the initial ID
must occur in `products`. Store-sourced title, price, period, availability, and
purchase state are intentionally not embedded in the document.

### `purchaseButton`

Displays its localized `label` and semantically requests purchase of the
currently selected product when activated. The purchase contract and result
types belong to the consuming SDK, not to executable protocol content.

### `restoreButton`

Displays its localized `label` and semantically requests purchase restoration
when activated.

### `closeButton`

Displays its localized `label` and semantically requests dismissal when
activated.

### `legalText`

Displays its localized `value` as legal copy. Protocol `0.1` does not model
links or rich text.

## Compatibility metadata

The compatibility manifest lists every capability defined by schema `0.1` and
its exact capability version. Each document lists the subset it actually uses
under `compatibility.requiredCapabilities`. Validation requires that:

1. every used layout and component capability is declared exactly once;
2. no unused capability is declared;
3. the manifest supports every declared capability at the exact version; and
4. the manifest includes every capability the schema can express.

All `0.1` capability versions are `0.1`. Readers must check the document's
schema version and required capabilities before decoding component payloads.

## Unsupported and unknown input

Protocol `0.1` uses atomic rejection:

| Condition                         | Required behavior             |
| --------------------------------- | ----------------------------- |
| Unknown schema version            | Reject the complete document. |
| Unknown component                 | Reject the complete document. |
| Unknown property                  | Reject the complete document. |
| Missing or unsupported capability | Reject the complete document. |
| Invalid component payload         | Reject the complete document. |

A reader must not partially render a rejected document. Its configuration
resolution layer may then select a previously validated cached document or a
bundled fallback. If neither exists, it returns a safe unavailable result.
Diagnostics should identify the rejected schema version, capability, component,
or property without exposing sensitive application data.

Component-level fallback is deliberately absent because every component in the
canonical `0.1` fixture is necessary to preserve its purchase and dismissal
semantics. Later optional components require a reviewed fallback contract before
they enter the schema.

## Reader integration

Each first-party SDK should use this sequence in its fixture test target:

1. Load `protocol/fixtures/v0.1/minimal-paywall.json` from the canonical path or
   a build resource produced from that path.
2. Verify that `schemaVersion` is exactly supported.
3. Verify every required capability and its exact version.
4. Decode the closed document and component models.
5. Verify the selected product reference during decoding.
6. Return a typed validation or unavailable result on any failure; do not
   partially decode or render.

SDK-specific model naming may remain idiomatic, but JSON field names and
component semantics must remain identical. The canonical fixture must decode
without SDK-specific configuration changes.

## Deliberately deferred

The following remain outside `0.1` and require protocol review before addition:

- additional or nested layout types
- scrolling and safe-area behavior
- images, assets, icons, badges, video, and custom components
- design-token and style catalogs
- visibility conditions, generic actions, and animation metadata
- rich legal links
- product interpolation and unavailable-product presentation
- localization catalogs, plurals, interpolation, and bidirectional layout
- component-level fallback behavior
