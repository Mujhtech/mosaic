# Mosaic Protocol 0.1 RC1

## Status and Phase 1 boundary

Protocol `0.1` RC1 is the approved Mosaic Phase 1 baseline. The product owner
accepted the unchanged RC1 bytes on 2026-07-17, and they are now immutable.

The candidate proves one complete local paywall contract across Flutter,
SwiftUI, and Jetpack Compose. It includes only native local rendering, bundled
fallback resolution, and commerce supplied by an injected mock provider. It
does not include remote configuration, REST fetching, Studio editing,
analytics, executable content, arbitrary style systems, real billing, or any
Phase 2 feature.

Canonical artifacts:

- schema: `protocol/schema/v0.1/paywall.schema.json`
- compatibility manifest: `protocol/compatibility/v0.1.json`
- fixture: `protocol/fixtures/v0.1/complete-paywall.json`

The fixture is the only complete canonical RC1 paywall. SDK builds may copy it
as a generated bundle resource, but the copy must be produced from that path.
No platform may maintain a hand-edited schema or fixture fork.

## Closed document contract

A document is a closed JSON object with these required fields:

| Field | Meaning |
| --- | --- |
| `schemaVersion` | Exact protocol version; RC1 accepts only `0.1`. |
| `id` | Stable paywall identifier. |
| `revision` | Integer from 1 through 2,147,483,647. |
| `compatibility` | Exact capabilities used by this document. |
| `localization` | Locale catalogs, fallback locale, and direction. |
| `assets` | Logical bundled image declarations and their fallbacks. |
| `products` | Provider-neutral product references. |
| `layout` | One root vertical scroll container. |

Every protocol object rejects unknown properties. `revision` uses JSON
Schema's mathematical integer rule, so `1.0` is valid JSON with an integral
value. Non-finite runtime values are not JSON and readers must reject them.
Publication immutability is outside this local-renderer schema.

## Logical layout semantics

### Units and direction

Protocol layout numbers are density-independent logical units:

| Renderer | One protocol logical unit |
| --- | --- |
| Flutter | one Flutter logical pixel |
| SwiftUI | one point |
| Jetpack Compose | one `dp` |

The mapping is semantic, not a physical-pixel guarantee. Font metrics,
anti-aliasing, native control chrome, scroll physics, focus indication, and
safe-area sizes remain native. Text continues to honor the host's accessibility
text scale and must not be clipped to preserve a fixed snapshot.

`start` and `end` are direction-relative. They map to leading/trailing in LTR
and reverse in RTL. The protocol deliberately has no `left` or `right` value.

### `scrollContainer`

The document root has:

- `axis: "vertical"`;
- `safeArea: "respect"`;
- `showsIndicators`, which controls indicator visibility; and
- one `content` vertical stack.

The viewport stays inside all host safe-area insets. Stack padding is applied
in addition to those insets. Content scrolls vertically when its measured
height exceeds the safe viewport. Source order is reading and focus order.
Nested scroll containers are not valid RC1 content. Native overscroll and
deceleration effects may differ and are not conformance failures.

### `verticalStack`

A vertical stack fills its available width, measures children in source order,
and has:

- `spacing`: space only between adjacent children;
- `padding`: nonnegative `top`, `start`, `bottom`, and `end` insets;
- `horizontalAlignment`: `start`, `center`, `end`, or `stretch`; and
- one or more `children`.

`stretch` assigns each child the available content width. Other alignments keep
the child's intrinsic width and position it along the cross axis. A vertical
stack may contain another vertical stack, so validation and rendering must walk
the tree recursively. A stack cannot contain another scroll container in RC1.

## Localization and bidirectionality

Visible and accessibility strings use:

```json
{
  "default": "Continue",
  "localizationKey": "paywall.purchase"
}
```

Locale tags use the canonical RC1 BCP-47 subset: a lowercase two- or
three-letter language plus an optional uppercase two-letter or three-digit
region, for example `en`, `ar`, `de-DE`, or `es-419`.

For a requested locale, a reader resolves text in this exact order:

1. exact requested tag, when declared;
2. requested base language, when declared;
3. `fallbackLocale`;
4. `defaultLocale`;
5. the localized value's inline `default`.

When no locale is requested, `defaultLocale` precedes `fallbackLocale`.
Repeated candidates are visited once.

Layout direction is resolved independently using the first declared locale in
the same candidate chain. A missing Arabic string may therefore fall back to
English text while the layout correctly remains RTL. Each catalog explicitly
declares `ltr` or `rtl`; readers must not guess direction from platform locale
APIs.

Semantic validation requires:

- declared catalogs for `defaultLocale` and `fallbackLocale`;
- every localized value's key in the default catalog;
- every inline `default` to equal its default-catalog value;
- no unused key in the default catalog; and
- no translated key absent from the default catalog.

Secondary catalogs may omit a key to exercise fallback. The canonical fixture
contains complete English, a deliberately long German path, and Arabic RTL in
the same document; these are locales of one fixture, not separate fixtures.
Plural rules and interpolation are outside RC1.

## Assets and image fallback

RC1 assets are image declarations with:

- a stable asset `id`;
- `source.type: "bundled"`;
- a logical, path-free `source.key`; and
- a required localized `placeholder` fallback.

The key is handed to a platform asset resolver. It is not a file path, URL, or
remote-fetch instruction. Each host maps the same key to a resource packaged
for that platform.

An `image` component declares:

- `assetId`;
- `width: "fill"`;
- a positive `aspectRatio` interpreted as width divided by height;
- `contentMode: "fit"` or `"fill"`; and
- informative or decorative accessibility metadata.

The image's allocated height is `allocated width / aspectRatio`. `fit`
preserves the entire source within that frame; `fill` preserves aspect ratio
while cropping overflow to cover it. If lookup or decoding fails, the renderer
shows the localized placeholder in the same allocated frame. This keeps native
layout deterministic without embedding a bitmap or platform path in the
protocol.

## Provider-neutral products

Each top-level product reference has:

- `id`: the document-local identifier used by components and outcomes;
- `productId`: an opaque identifier passed unchanged to the injected purchase
  provider;
- a localized marketing `label`; and
- an optional localized `badge`.

Product reference IDs and provider product IDs are both unique. The protocol
does not contain localized price, currency, subscription period, trial,
introductory offer, entitlement, or availability. Those values come from the
runtime purchase provider. RC1 does not name RevenueCat, StoreKit, or Google
Play and mock commerce remains an SDK concern.

## Component semantics

### `text`

Displays localized `value` with semantic `style` `title`, `body`, or `caption`
and direction-relative `alignment` `start`, `center`, or `end`. Accessibility
declares either ordinary `text` or `heading` with a level from 1 through 6.
Renderers map semantic styles to native text styles and honor text scaling.

### `image`

Resolves its declared image asset and uses the deterministic frame and fallback
rules above. Accessibility is exactly one of:

- `{ "hidden": true }` for decorative content; or
- `{ "hidden": false, "label": localizedText }` for informative content.

An informative image without a label and a decorative image with a label are
both invalid.

### `featureList`

Renders one or more items in source order with `marker: "checkmark"` and
logical `itemSpacing`. Each item has a stable local ID and localized text. The
component supplies a localized accessibility group label; items remain
individually readable in source order. Marker artwork may be platform-native.

### `productSelector`

Renders `productReferenceIds` in source order, initially selecting
`initiallySelectedProductReferenceId`. Each available option exposes the
localized protocol label, optional badge, purchase-provider price/period data,
and native selected state.

Unavailable products follow one exact RC1 policy:

1. omit unavailable product options;
2. retain the configured initial selection when it is available;
3. otherwise select the first available reference in source order;
4. when none are available, clear selection, show the localized fallback
   message, disable every purchase action targeting the selector, and emit the
   `productUnavailable` normalized notification.

The selector has a localized accessibility label and optional hint. Each
option must expose selected/not-selected and enabled state through native
semantics.

### `purchaseButton`

Displays `label`, replaces it with `inProgressLabel` while one purchase is in
flight, and executes its closed purchase action against the referenced product
selector. It is disabled when the selector has no available selection or while
the same action is in flight. It has a localized accessibility label and
optional hint and exposes native enabled and busy state.

### `restoreButton`

Displays `label`, replaces it with `inProgressLabel` during restore, and
executes the closed restore action. It has a localized accessibility label and
optional hint and exposes native enabled and busy state.

### `closeButton`

Displays `label`, executes the closed close action, and has a localized
accessibility label and optional hint. Its visual treatment may use native
button chrome, but the visible label and accessible action remain present.

### `legalText`

Displays localized static legal `value` with direction-relative alignment and
ordinary text accessibility. RC1 has no rich text, link action, or hidden
legal-copy behavior.

## Declarative actions

RC1 has only three closed actions:

```json
{ "type": "purchase", "productSelectorId": "plans" }
{ "type": "restore" }
{ "type": "close" }
```

A purchase action must resolve to a product selector in the same recursive
layout tree. Every selector must have at least one purchase action. Action
objects reject unknown properties and cannot contain scripts, URLs, expressions,
or platform callbacks.

## Normalized outcomes

The compatibility manifest is the machine-readable source for exact outcome
spelling and ordering.

Interaction outcomes:

| Interaction | Outcomes |
| --- | --- |
| Select product | `productSelected` |
| Purchase | `purchased`, `alreadyEntitled`, `cancelled`, `productUnavailable`, `purchaseFailed` |
| Restore | `restored`, `alreadyEntitled`, `restoreNoPurchases`, `restoreFailed` |
| Close | `dismissed` |

An outcome concerning a product carries its document-local product reference
ID. Failure outcomes may carry a safe SDK diagnostic code, never provider
credentials or raw internal errors.

The host-facing presentation outcome union is exactly:

- `purchased`
- `restored`
- `alreadyEntitled`
- `dismissed`
- `cancelled`
- `productUnavailable`
- `configurationUnavailable`
- `purchaseFailed`
- `renderingFailed`

`restoreNoPurchases` and `restoreFailed` are intentionally interaction-only.
They leave the paywall presented so the user may retry, purchase, or close;
they must not be misreported as `cancelled` or `purchaseFailed`. A successful
restore maps to `restored`, and a restore that confirms existing access maps to
`alreadyEntitled`.

`configurationUnavailable` means neither the requested document nor a valid
bundled fallback could be resolved. `renderingFailed` means a validated,
compatible document encountered an unrecoverable renderer error. Product
selection itself never ends a presentation.

The protocol normalizes outcome meaning and payload identity. Each SDK may use
idiomatic enum/sealed-type names and callback or async APIs, but its JSON-facing
and conformance values must map one-to-one to these strings. Modal dismissal
and embedded-view ownership remain SDK API concerns and must be stated by each
SDK rather than inferred from the protocol.

## Compatibility metadata

Every capability has exact version `0.1`. RC1 defines:

- `layout.scrollContainer`, `layout.verticalStack`;
- one `component.*` capability for each of the eight component types;
- `localization.catalogs`, `localization.rtl`;
- `product.references`, `asset.bundledImage`;
- `action.purchase`, `action.restore`, `action.close`;
- `accessibility.metadata`;
- `fallback.asset`, `fallback.product`; and
- `outcome.normalized`.

Document capability derivation walks the complete tree, including nested
vertical stacks. It also derives capabilities from locale direction, catalogs,
assets, products, accessibility records, fallbacks, and actions. Validation
requires every used capability exactly once, rejects unused declarations,
requires an exact manifest version match, and requires the manifest to list
every capability the schema can express.

First-party SDK capability reports must declare SDK version, supported schema
versions, and every supported capability/version pair. RC1 defines no custom
component capability.

## Safe failure and bundled fallback

Protocol validation is atomic:

| Condition | Required behavior |
| --- | --- |
| Unknown schema version | Reject the document. |
| Unknown component or property | Reject the document. |
| Missing/unsupported capability | Reject the document. |
| Duplicate ID or invalid reference/action | Reject the document. |
| Invalid locale, product, asset, or accessibility payload | Reject the document. |
| Primary document rejected | Attempt the configured bundled fallback. |
| Bundled fallback rejected | Return `configurationUnavailable`. |
| Bundled image missing/undecodable | Use its declared localized placeholder. |
| Product unavailable | Apply the selector fallback described above. |

A reader must never partially render an invalid or unsupported document. For
Phase 1 the canonical fixture itself is the shared bundled fallback document.
Its loading mechanism belongs to the SDK; the document does not contain a file
path or network request.

## Semantic validation beyond JSON Schema

The protocol validator additionally enforces:

- globally unique layout/component IDs and list-local feature IDs;
- unique asset IDs, product reference IDs, and provider product IDs;
- valid and used asset/product declarations;
- valid initial selection and purchase-selector action references;
- at least one purchase action for every selector;
- default/fallback locale existence and catalog consistency;
- exact inline-default/default-catalog equality;
- recursive capability derivation and manifest completeness;
- canonical RC1 coverage for every component, nested stack, RTL, and a long
  translation; and
- canonical two-space JSON with a final newline for every JSON artifact.

## Allowed native differences

Cross-platform conformance is semantic rather than pixel identity. Native
typography metrics, button chrome, checkmark glyphs, focus visuals, safe-area
inset sizes, scroll physics, and image decoding may differ. The following may
not differ: component/source order, logical spacing and direction, image frame
aspect, locale/fallback resolution, product availability/selection rules,
action targets, accessibility roles/labels/state, bundled fallback decisions,
and normalized outcomes.

## Deliberately deferred

RC1 does not define horizontal stacks, arbitrary containers, fixed text
heights, design-token catalogs, rich text links, interpolation, plurals,
animations, conditions, custom components, remote assets, remote configuration,
analytics, Studio metadata, cached configuration, real purchase-provider
adapters, or executable content. Adding any of these requires protocol review
and cannot be smuggled into an SDK-specific model.
